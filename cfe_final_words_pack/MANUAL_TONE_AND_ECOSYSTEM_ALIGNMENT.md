# MANUAL_TONE_AND_ECOSYSTEM_ALIGNMENT

This file defines the final language standard for CFE so it feels like FPE’s sibling in the same ecosystem.

## Voice principles
CFE should sound:
- serious
- calm
- exact without pretending omniscience
- campaign-literate
- dense enough to teach
- direct enough to be usable under pressure

It should not sound:
- like generic SaaS copy
- like lightweight startup productivity software
- like consultant fluff
- like overexcited “AI insight” language
- like a compliance disclaimer farm
- like a toy dashboard

## Relationship to FPE
CFE should feel like it was built by the same system designer who built FPE:
- same respect for canonical calculations
- same respect for operator judgment
- same preference for dense, useful explanation over decorative text
- same refusal to hide uncertainty with fake smoothness
- same right-rail/manual teaching standard
- same emphasis on practical decision support

The apps do not need identical wording, but they should feel like siblings:
- shared seriousness
- shared discipline
- shared visual restraint
- shared use of warnings as decision support rather than drama
- shared respect for human operators

## Manual density standard
The manual is a product surface, not a help afterthought.

A good CFE manual section:
- teaches what the page is for,
- defines what the numbers mean,
- explains what healthy and weak patterns look like,
- warns against common misreadings,
- and gives practical next-move guidance.

A weak manual section:
- just repeats labels,
- says “this page shows…”
- avoids judgment,
- or assumes the operator already knows why the page matters.

## Report language standard
Reports should sound circulation-ready:
- not too casual,
- not legalistic,
- not bloated,
- not bland.

Each report should make it easy for the reader to answer:
- what condition are we in,
- why,
- what does it threaten,
- what should happen next.

## Warning language standard
Warnings should be:
- specific,
- non-hysterical,
- tied to an actual trigger,
- useful immediately.

Each warning should contain:
- condition,
- consequence,
- recommended action.

## Right-rail standard
The right rail is not decorative. It is where the application teaches in real time.

Each right-rail block should:
- tell the operator how to read the page,
- note one common mistake,
- note one best practice,
- help interpret caution without melodrama.

## Final rule
When there is a choice between sounding sleek and sounding useful, choose useful.
When there is a choice between sounding minimal and teaching clearly, choose clear.
When there is a choice between false certainty and explicit uncertainty, choose explicit uncertainty.
